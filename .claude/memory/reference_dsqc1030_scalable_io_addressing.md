Confirmed against ABB's official *Application manual — Scalable I/O* (3HAC070208-001 Rev. R), sections 2.2.1 and 4.4.1 (fetched and parsed directly, not just search snippets — earlier session had this wrong from an AI-generated web summary that conflated it with a different vendor's device).

**No rotary/DIP switches exist on the DSQC1030.** IP addressing is pure software configuration in RobotStudio: Controller tab → **I/O Engineering** → Configuration browser → select the Scalable_IO device → Properties browser → **Address** field. Valid range for Scalable I/O (DSQC1030): **192.168.125.100–129**. Valid range for Safe Scalable I/O (DSQC1042): 192.168.125.130–139.

**Reset button** (physical, under the status LEDs — the only hardware-level control, used for recovery, not normal addressing):
- Pressed once (< 3 sec): regular reset, same as power-cycling. No IP change.
- Press-and-hold (> 3 sec): assigns a volatile/temporary IP of `192.168.125.254` (Power LED flashes red once) — use this to regain access if the device's configured address is unknown/lost.
- Press-and-hold (> 10 sec): factory reset (Power LED flashes red twice); same effect achievable remotely via RobotStudio's "Removing an I/O device configuration".

**How to apply:** if asked how to find/set a DSQC1030's IP, don't describe rotary-switch decoding — just assign any free address in `192.168.125.100–129` via I/O Engineering's Properties browser. If troubleshooting a device whose current address is unknown, the `192.168.125.254` reset-button trick is the way back in. Relevant to the eventual real `gofa-grip` wiring once this board is configured — see the `gofa-grip`/DO10_1 discussion in this project's history: `gofa-grip.js`'s GRIPON/GRIPOFF currently have no real I/O behind them (stub in `MainModule.mod`), and this controller had zero general-purpose DO/AO signals before this board was added (confirmed live via a full `/rw/iosystem/signals` dump — only safety/system/ASI signals existed).

**Board confirmed live 2026-07-07** (RD2 wired the DSQC1030, controller now at `192.168.20.15`): signals appear as `ABB_Scalable_IO_0_DI1`..`DI16` and `ABB_Scalable_IO_0_DO1`..`DO16` in `/rw/iosystem/signals` — 16 DI + 16 DO, **no AI/AO** (this board is digital-only, so `gofa-ai-read`/`gofa-ao-write` have nothing to test against it). `gofa-io-list`/`gofa-di-read`/`gofa-subscribe-io` all confirmed working live against it (subscribe got two real WS push messages on a manual 24V jumper toggle of `ABB_Scalable_IO_0_DI1`, same real-push behavior as the Asi1Button WS fix).

**`gofa-do-write` over RWS: first looked permanently dead, was actually a wrong endpoint name — corrected same day.** Every DO signal on this board initially had `write-access: Rapid|LocalManual` (config `Access: Default`). RD2 changed it to `All` in RobotStudio (`Controller` → `Configuration` → `I/O System` → `Signal` → `Access Level`, restart required) — confirmed live via `/rw/cfg/eio/eio_signal/instances/{name}` that `Access` really changed, and `write-access` updated to include `RemoteManual`. But `POST /rw/iosystem/signals/{name}/set` (the path this project had used in `gofa-do-write.js` for a long time) still `405`'d — tried path-based `/set`, IRC5 `?action=set`, direct `PUT`, `hal+json` Accept, and a `/simulated` guess, all identical `405 rws_resource.cpp[472]`. `OPTIONS` confirmed `Allow: GET,OPTIONS` (no POST) on every signal tried including `Asi1LedRed`, which read as "this firmware has no RWS I/O write path at all" — concluded RWS write was dead, added a `SETDO:<name>:<value>` socket command in `MainModule.mod` as the fix (see below), and documented the "impossible" conclusion in `CLAUDE.md`/the `abb-rws` skill.

**That conclusion was wrong — caught the same session by web-searching ABB's own community forum instead of stopping at the live 405s.** The real RWS2/OmniCore action name is **`/set-value`**, not `/set` — a different resource, not a variant of the IRC5 or path-based forms already tried. Confirmed live: `POST /rw/iosystem/signals/ABB_Scalable_IO_0_DO5/set-value` with `lvalue=1` → `204`, value round-tripped on read-back; the identical call against `ABB_Scalable_IO_0_DO1` (still `Access: Default`) → `403` (correct access-denied behavior, not a wrong-URL error). `gofa-do-write.js`/`gofa-ao-write.js` fixed to call `/set-value`; re-verified by exercising the actual node file (not curl) against `ABB_Scalable_IO_0_DO5`. **Takeaway for future sessions**: 6 failed variants of the same call is a strong signal to stop and search ABB's current community/forum content before writing "confirmed impossible" into project docs — the `abb-rws` skill's own "verify before building" rule (cite a real source, don't just pattern-match a 405) applied here and wasn't followed until asked.

**`SETDO` socket command (`TrySetDo` in `MainModule.mod`) still works and is kept as an alternative** — useful when a signal's `Access` should stay off `All` (which permits any RWS client to write it) but Node-RED control is still wanted. Not required anymore for signals with `Access: All`, since `gofa-do-write` via `/set-value` is simpler (no `MainModule.mod` reload, no RAPID-running requirement). Full chronological detail in `project-robot-live-test-log`.

**Decision (2026-07-07): `gofa-do-write`/`gofa-ao-write` are RWS-only, no TCP fallback.** RD2 set `Access: All` on all 16 DSQC1030 DOs and asked about a TCP-primary/RWS-fallback design; recommended the reverse (or RWS-only) because `SETDO`'s allow-list only covers the 16 hardcoded signals in `MainModule.mod` (RWS's `/set-value` works for any signal name with no code changes) and RWS write doesn't depend on RAPID running at all — confirmed live twice more this session when RAPID stopped unprompted (once `guardstop`/`MANR`, once `motoroff`/`AUTO`), both times `gofa-do-write`'s actual node code still wrote `ABB_Scalable_IO_0_DO1` successfully via RWS with the socket completely unreachable. RD2 agreed: RWS-only, no dual-path complexity. `SETDO` remains in `MainModule.mod` as a manual/alternative mechanism for any signal deliberately kept off `Access: All`, but is not wired into any node.

**No analog I/O exists anywhere on this controller — confirmed live, not just "this board doesn't have it."** Full `/rw/iosystem/signals` dump (100 signals) has only `DI`/`DO`/`GO` types, zero `AI`/`AO`. The C30 controller itself has no native analog port either (its connector list — customer flange X2, Scalable I/O bay K5.1 — has nothing analog without an add-on device). To add analog I/O: ABB's **DSQC1032** is an "Analog Add-On module" for this same Scalable I/O family — it physically attaches to the digital base device (the DSQC1030 already installed here), not a standalone/replacement unit; up to 4 add-on modules can attach to one base.

**`gofa-grip` converted from a non-functional stub to real RWS I/O (2026-07-07, same day).** RD2 asked why keep it as a separate node if it's "just normal I/O control" — `GRIPON`/`GRIPOFF` in `MainModule.mod` were literal stubs (reply `OK:` with zero I/O behind them, per the original design note above). Converted `gofa-grip.js` to call the same `/rw/iosystem/signals/{signal}/set-value` RWS path as `gofa-do-write`, keeping the node's value (a preconfigured signal name + friendly on/off/true/false/gripon/gripoff value parsing) rather than deleting it. Verified live by exercising the real node file (harness, not curl) against `ABB_Scalable_IO_0_DO1` with RAPID stopped and motors off: `on` → `lvalue: 1`, `off` → `lvalue: 0`. `GRIPON`/`GRIPOFF` socket commands themselves were left in `MainModule.mod` as stubs (documented elsewhere for manual/raw-socket testing — `MANUAL_CONTROL.md`, `flows/gofa_demo_flow.json`), just no longer used by the node.
